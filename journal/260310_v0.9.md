## 1. VLM 벤치마크 결과 (OI-01)

### 환경
- Jetson Orin NX 16GB, JetPack 6.2, CUDA 12.6
- `dustynv/nano_llm:r36.4.0` 컨테이너 내부
- 모델: `Efficient-Large-Model/VILA1.5-3b` (MLC backend, `q4f16_ft` 양자화)
- 첫 실행: MLC 컴파일 ~197s / 이후 실행: ~10s (캐시 `/data/models/mlc/`)

### NanoLLM 올바른 API 사용 (멀티모달 추론)

**잘못된 방법** (ValueError: inhomogeneous shape):
```python
model.generate([img, PROMPT])
```

**올바른 방법** (ChatHistory API):
```python
chat = ChatHistory(model)
chat.append('user', image=img)
chat.append('user', text=prompt)
embedding, _ = chat.embed_chat()
for token in model.generate(embedding, max_new_tokens=64, streaming=True):
    ...
```

- `chat_history.reset()` 후 반드시 `gc.collect()` 호출 필요 (NanoLLM GitHub issue #39, 메모리 누수 방지)

### 벤치마크 수치 (VILA1.5-3b, 384×384 랜덤 프레임)

| 프레임 수 | 평균 추론 시간 | 비고 |
|---|---|---|
| 1 | 1,066 ms | ~1s/frame |
| 4 | 4,027 ms | 선형 비례 |
| 8 | 7,522 ms | 선형 비례 |

- GPU 메모리 사용량: **7.4 GB**
- 추론 시간은 프레임 수에 거의 선형 비례 → 단일 프레임 반복 처리 구조로 설계

> OI-01 잠정 결정: **VILA1.5-3b** (7.4GB RAM, ~1s/frame). 3B급 정확도 검증 후 필요 시 상위 모델 검토.

## 2. 프레임 샘플링 설계 확정 (README v0.8)

### 문제 제기

- 기존 설계 (VLM 준비 시 최신 프레임 취득): 카메라 FPS에 따라 추론 타이밍이 변동. 30/60 FPS 카메라 연결 시 간격 불안정.
- VLM 처리 중 프레임이 drop되는 것은 불가피하나, **원하는 간격으로 샘플링**하는 것은 가능.

### 채택 방안 (Option B): videorate를 통한 FPS 정규화

```
nvvidconv → videorate (target_fps) → appsink → Python (N프레임 누적) → VLM
```

- `target_fps = N_frames / inference_interval` 로 설정 (예: 1fps×30s=30프레임)
- 카메라 FPS 무관하게 균일한 간격으로 프레임 추출
- VLM 추론 중 videorate가 버퍼 관리 → Python 측 로직 단순화

## 3. 링버퍼 역할 재정의 (README v0.9)

### 문제 제기

- 이전 설계: 링버퍼에 H.264 클립을 저장하여 이벤트 발생 시 저장 (OI-06)
- 충돌: 링버퍼를 VLM 추론용(프레임 누적)과 클립 저장용으로 이중 사용하려 했으나 목적 불일치

### 최종 설계 분리

| 역할 | 담당 |
|---|---|
| VLM 추론용 프레임 보관 | Python Ring Buffer (1fps, 최대 N초분) |
| 이벤트 클립 저장 | **MediaMTX 세그먼트 레코딩** |

**MediaMTX 세그먼트 레코딩 (`config/mediamtx.yml`)**:
```yaml
record: yes
recordPath: /recordings/%path/%Y-%m-%d_%H-%M-%S
recordFormat: fmp4
recordSegmentDuration: 60s
recordDeleteAfter: 2h
```

- MediaMTX가 60초 단위로 연속 저장
- 이벤트 판정 시 App이 해당 시각의 세그먼트 파일을 `preserved/`로 복사 (삭제 방지)
- 링버퍼는 VLM 전용으로 단순화: 1fps × 30s = 30프레임 ≈ 237MB (RGBA)

> OI-06 방향 확정: MediaMTX 세그먼트 레코딩 + App에서 preserve

## 4. App 컨테이너 Dockerfile 완성

### 최종 방식: 런타임 bind-mount

**문제**: `nvidia-l4t-core` pre-install 스크립트가 `/proc/device-tree/compatible` 읽기 시도 → Docker 빌드 컨텍스트에 없어 실패

**해결**: NVIDIA GStreamer 플러그인을 빌드 시점에 설치하지 않고, 런타임에 호스트 디렉토리를 마운트

### `docker/app/Dockerfile` (최종)

```dockerfile
FROM dustynv/nano_llm:r36.4.0
# gstreamer1.0-plugins-bad만 빌드 시 설치 (h264parse, rtph264depay, videorate 등)
# NVIDIA 플러그인(nvv4l2decoder, nvvidconv 등)은 런타임 마운트
RUN apt-get update && apt-get install -y --no-install-recommends \
        gstreamer1.0-plugins-bad \
    && rm -rf /var/lib/apt/lists/*
RUN pip install --no-cache-dir pillow numpy
WORKDIR /app
COPY src/app/ .
CMD ["python3", "main.py"]
```

### `compose.yml` 런타임 마운트 설정

```yaml
environment:
  - GST_PLUGIN_PATH=/usr/lib/aarch64-linux-gnu/gstreamer-1.0
  - LD_LIBRARY_PATH=/usr/lib/aarch64-linux-gnu/nvidia:/usr/lib/aarch64-linux-gnu
volumes:
  - /usr/lib/aarch64-linux-gnu/gstreamer-1.0:/usr/lib/aarch64-linux-gnu/gstreamer-1.0:ro
  - /usr/lib/aarch64-linux-gnu/nvidia:/usr/lib/aarch64-linux-gnu/nvidia:ro
```
