주제: HLS/WebRTC 스트리밍 인증 — MediaMTX JWT 패턴
작성일: 2026-06-11
관련 섹션: 2.2, 2.3 (재설계 시 반영 필요)

================================================================================
배경
================================================================================

재설계 목표: 모든 Client App 요청이 API Server를 단일 진입점으로 거치도록 한다.
문제: HLS/WebRTC 스트리밍은 대용량 데이터이므로 API Server가 중계하는 것은
      부하와 복잡도 측면에서 부적합하다.

================================================================================
해결 패턴: Token-based Redirect (Signed URL / Playback Token)
================================================================================

흐름:
  ① Client App → API Server   : 스트림 URL 요청 (인증 포함)
  ② API Server → Client App   : 단기 유효 JWT 발급
  ③ Client App → MediaMTX     : JWT를 포함하여 HLS/WebRTC 직접 연결 요청
  ④ MediaMTX                  : JWT 자체 검증 후 스트림 송신 (Client App ← MediaMTX)

핵심:
- 제어 경로(control plane): Client App → API Server (인증/인가, 토큰 발급)
- 데이터 경로(data plane): Client App ↔ MediaMTX (실제 스트리밍)
- "단일 진입점" 원칙은 제어 경로에서 지켜진다.
  스트림 URL과 JWT는 반드시 API Server를 통해서만 발급되므로,
  API Server가 취소하거나 JWT가 만료되면 스트리밍도 차단된다.

================================================================================
MediaMTX JWT 지원 여부
================================================================================

공식 지원 확인됨 (mediamtx.org/docs/usage/authentication).

방식 A — JWT 직접 검증 (authMethod: jwt)
  - MediaMTX가 API Server의 JWKS 엔드포인트에서 공개키를 가져와 자체 검증
  - mediamtx.yml 설정 예시:
      authMethod: jwt
      authJWTJWKS: https://your-api-server/api/auth/jwks.json
  - 클라이언트 전달 방법:
      Authorization: Bearer <JWT>  (권장)
      또는 ?jwt=<JWT> 쿼리 파라미터 (비권장 — 로그 노출 위험)

방식 B — HTTP 콜백 (authMethod: http)
  - MediaMTX가 클라이언트 요청마다 API Server에 인증 위임
  - 더 단순하지만 API Server에 요청 부하 발생

주의사항:
  - JWKS는 시작 시 한 번만 가져오므로 키 교체 시 MediaMTX 재시작 필요
  - v1.8.0 이전에는 Authorization 헤더 JWT 무시 버그 있음 (v1.8.0에서 수정)

업계 사례
================================================================================

동일 패턴을 사용하는 서비스:
  - AWS IVS: "Playback Token" (ES384 JWT)
    https://docs.aws.amazon.com/ivs/latest/LowLatencyUserGuide/private-channels-workflow.html
  - Cloudflare Stream: "Signed URL" (RS256 JWT)
    https://developers.cloudflare.com/stream/viewing-videos/securing-your-stream/
  - Mux: "Signed URL"
    https://www.mux.com/articles/securing-video-playback-with-signed-urls

패턴 명칭: "Signed URL", "Playback Token", "Token Vending Architecture" 등
           단일 공식 명칭은 없으나 업계 표준 패턴으로 확인됨.
