## 6.1 조사 기록; Findings

결정에 앞서 확인된 사실을 기록한다. 결정이 내려지면 위 표로 이관한다.

### F-1. App Server 8080 호스트 게시 (D-A 관련)

App Server의 8080 포트가 `docker-compose.yml`에서 호스트에 게시되어 있어, 클라이언트가 API Server를 우회하여 App Server에 직접 접근하는 경로가 외부에 열려 있다. 이는 의도된 운영 설계가 아니라 **디버그 잔재**이다.

- **근거(이력).** 8080 게시는 커밋 `be96082`("디버그 대시보드…")에서 주석 `# Debug web UI`와 함께 도입되었다. 즉 App이 직접 서빙하는 디버그 대시보드(`/stream` MJPEG, `/events` SSE 등)에 API를 거치지 않고 붙기 위한 것이었다. 이후 커밋 `b11177a`에서 주석만 `# App HTTP API`로 변경되었고, **호스트 게시는 회수되지 않은 채 상존한다.**
- **정상 대비.** 같은 파일의 MediaMTX RTSP 8554는 의도적으로 `ports`에 넣지 않아 내부망(`mtx:8554`) 전용이다. App 8080의 정당한 소비자도 API 하나뿐이며 내부 주소(`app:8080`)로 충분하다.
- **함의.** "API Server 단일 진입점"(SRS 2.7, D-A) 원칙과 코드가 어긋나 있음을 보여주는 구체적 증거이다. D-A를 단일 진입점(패턴 A)으로 정리할 경우 8080의 호스트 게시는 제거 대상이다. 단, 스트리밍·SSE 경로가 API 프록시를 거쳐도 되는지 먼저 확정해야 하므로(D-B 연동), 제거 순서는 **D-A(스트리밍 경로) 결정 → 8080 포트 처리**이다.
