## 2. 항목별 결정 사항

### (1) Branch A — 패스스루 구조 확정

- 기존 설계: `rtspsrc → DEC → TEE → NVENC → MediaMTX` (불필요한 재인코딩)
- **확정 구조**: `rtspsrc → TEE → (Branch A) 패스스루 → MediaMTX`
- Branch A는 인코딩 스트림 그대로 포워딩 → GPU 사용 없음
- Branch B는 TEE 이후 디코딩

```
rtspsrc → tee ┬─ (Branch A) queue → [pass-through] → MediaMTX
              └─ (Branch B) queue → nvv4l2decoder → nvvideoconvert → appsink
```

> OI-05 결정 시 "디코딩 전 tee" 구조임을 명시할 것.

---

### (2) Ring Buffer — 개념 정리 및 크기 결정 기준

- Ring Buffer = EvictingQueue 구조와 동일. 고정 크기, 가득 차면 가장 오래된 데이터부터 삭제.
- **두 가지 목적**이 있으며, 크기 결정 기준이 다름:

| 목적 | 크기 기준 |
|---|---|
| VLM 추론 입력 | 한 번의 추론에 넣을 프레임 수 (VILA 기준 최대 8프레임) |
| 소급 영상 저장 | 이상 감지 전 몇 초를 클립에 포함할지 (예: 10~30초) |

- 하나의 Ring Buffer로 통합 가능: 소급 저장 윈도우 크기로 유지, VLM 입력은 최근 N프레임만 슬라이싱
- **크기 = max(VLM 추론 윈도우, 소급 저장 윈도우)**
- OI-04 결정 시 "저장 포맷(RGB vs 인코딩)"과 "저장 FPS"도 함께 결정 필요

### (3) NanoLLM VILA 멀티프레임 분석 — 사실 확인 완료

- VILA 1.5-3B는 멀티프레임(비디오) 분석 지원 확인
- `vision/video.py`에서 rolling frame history 관리
- **Orin NX 16GB 기준 최대 8프레임 권장** (NX8 / Orin Nano는 4프레임)
- SigLIP 384×384 vision encoder: 프레임 1장 → ~192 토큰으로 압축

---

### (4) FCM 알림 수신 구조 — 개념 정리

택배 비유:
```
Jetson(발송인) → FCM 서버(택배사, Google) → 견주 스마트폰(수령인)
```

흐름:
1. 앱 설치 시 FCM에서 기기 토큰 발급 → Jetson API 서버에 등록
2. 이상 감지 → Jetson이 FCM HTTP v1 API 호출 (토큰 + 알림 내용)
3. FCM → 견주 폰 푸시 알림

> 기기 토큰 등록 수신은 API 서버(컨테이너)가 담당.

---

### (5) 컨테이너 구성 — 확정

API 서버 + DB를 Docker 컨테이너로 엣지에 포함. 개인 단일 사용자 환경이므로 부하 문제 없음.

| 컨테이너 | 역할 |
|---|---|
| NanoLLM | VLM 추론 (VILA 등) |
| MediaMTX | RTSP/WebRTC 스트리밍 (패스스루) |
| App | GStreamer 파이프라인, 추론 판정, FCM 발송 |
| API 서버 | 기기 토큰 등록, 이벤트 이력 조회, 영상 클립 제공 |
| DB | 이벤트 기록, 기기 토큰 저장 (SQLite 내장 or PostgreSQL 컨테이너) |

## 3. 기술 참고 사항

### VLM 해상도와 추론 시간의 관계

- SigLIP은 입력 해상도를 항상 384×384로 리사이즈 → **입력 해상도는 VLM 추론 시간에 영향 없음**
- 분석 브랜치 해상도를 낮추면: GStreamer 부하 감소 + Ring Buffer 메모리 감소 (권장)

| 해상도 | Ring Buffer 메모리(RGB) | VLM 추론 시간 |
|---|---|---|
| 1080p | ~6MB/frame | 동일 |
| 480p | ~1.3MB/frame | 동일 |
| 384p | ~0.8MB/frame | 동일 |

### VILA 1.5-3B 추론 시간 추정 (NX 16GB)

공식 벤치마크 없음. 확인된 사실로부터 추정:
- AGX Orin (275 TOPS): 단일 프레임 ~143ms (≈7 FPS)
- Orin NX 16GB (100 TOPS): 단일 프레임 ~400ms 추정
- 8프레임: ~1~3초/추론 추정 (실측 필요)

> Phase 0에서 `--max-images 8` 옵션으로 실측 필수 (OI-01, OI-02 결정의 핵심 실험)
