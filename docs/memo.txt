================================================================================
1부. 재설계 시 반드시 해소해야 할 경계 모호성 (구 comment.txt)
================================================================================

프로토타입(master 브랜치)에서 드러난 책임 경계 모호성 네 가지.
각 항목은 refactoring 브랜치 설계 과정에서 명시적으로 결정되어야 한다.

--------------------------------------------------------------------------------
1. 클립 파일 경로 스키마 소유권
--------------------------------------------------------------------------------

현황: `/data/{YYYY}/{MM}/{timestamp}.mp4` 경로 파싱 로직이
`api/clip_support.py`와 `app/server_support.py` 양쪽에 거의 동일하게 중복되어 있다.

결정 필요: App은 지정 위치에 파일을 쓰기만 하는가,
아니면 App이 클립 목록과 메타데이터 제공까지 책임지는가.

SRS 반영 상태: 역할 분담(App이 쓰고 API가 조회·삭제)은 SRS 2.2/2.3에 반영됨.
경로 스키마 소유권과 파싱 중복 해소는 미결.

--------------------------------------------------------------------------------
2. 카메라 설정(Apply)의 책임 소재
--------------------------------------------------------------------------------

현황: 브라우저 → API → (proxy) → App 순서로 카메라 설정이 전달된다.
하지만 응답 형성(누가 어떤 응답을 만드는가)과
파이프라인 재시작 타이밍(동기 vs 비동기)에 대한 명시적 계약이 없다.

결정 필요: API는 단순 프록시인가, 아니면 응답 형성에 관여하는가.
파이프라인 재시작은 App이 자율적으로 수행하고 API는 개입하지 않는다는
계약을 명시해야 한다.

SRS 반영 상태: 흐름은 SRS 2.3-A, FR-010에 반영됨. 응답 형성 주체와
재시작 동기/비동기 계약은 미결.

--------------------------------------------------------------------------------
3. App-API 간 내부 계약 (이벤트 알림)
--------------------------------------------------------------------------------

현황: App이 트리거 이벤트를 감지하면 클립을 저장하지만,
API의 SQLite 이벤트 로그에는 별도로 기록되지 않는다.
두 서비스는 공유 볼륨(/data)으로만 간접 연결되어 있고
명시적인 내부 API 계약이 없다.

결정 필요: App이 이벤트 발생 시 API를 직접 호출하는가,
아니면 API가 파일시스템 변경을 감지(또는 주기적으로 스캔)하는가.
또는 이벤트 로그 자체를 App이 소유하는가.

SRS 반영 상태: FR-023이 이벤트 이력 기록을 요구하나 기록 주체·메커니즘은 미결.

--------------------------------------------------------------------------------
4. SSE 필드 계약 (App → 클라이언트)
--------------------------------------------------------------------------------

현황: App의 `state.py`가 SSE 이벤트를 생산하고,
`web/useSSE.js`가 소비하지만, 필드 정의가 암묵적이다.
필드 추가·제거 시 양쪽을 수동으로 동기화해야 한다.

결정 필요: SSE 페이로드 스키마를 어디에 명시적으로 정의할 것인가.
또한 App의 내부 상태 중 어디까지를 클라이언트에 노출할 것인가의
경계를 결정해야 한다.

SRS 반영 상태: FR-028과 4.1의 보류 표에 항목만 존재. 스키마 정의 위치는 미결.

================================================================================
2부. HLS/WebRTC 스트리밍 인증 — MediaMTX JWT 패턴 (구 memo/mediamtx_jwt_streaming_auth.txt)
================================================================================

작성일: 2026-06-11
SRS 반영 상태: 패턴 자체는 SRS 2.3-C, IF-003, FR-015~017에 요구사항으로 반영됨.
아래의 MediaMTX 설정 구체사항·함정 정보·업계 사례는 설계/구현 시 참조.

배경
----
재설계 목표: 모든 Client App 요청이 API Server를 단일 진입점으로 거치도록 한다.
문제: HLS/WebRTC 스트리밍은 대용량 데이터이므로 API Server가 중계하는 것은
      부하와 복잡도 측면에서 부적합하다.

해결 패턴: Token-based Redirect (Signed URL / Playback Token)
------------------------------------------------------------
흐름:
  ① Client App → API Server   : 스트림 URL 요청 (인증 포함)
  ② API Server → Client App   : 단기 유효 JWT 발급
  ③ Client App → MediaMTX     : JWT를 포함하여 HLS/WebRTC 직접 연결 요청
  ④ MediaMTX                  : JWT 자체 검증 후 스트림 송신 (Client App ← MediaMTX)

핵심:
- 제어 경로(control plane): Client App → API Server (인증/인가, 토큰 발급)
- 데이터 경로(data plane): Client App ↔ MediaMTX (실제 스트리밍)
- "단일 진입점" 원칙은 제어 경로에서 지켜진다.
  스트림 URL과 JWT는 반드시 API Server를 통해서만 발급되므로,
  API Server가 취소하거나 JWT가 만료되면 스트리밍도 차단된다.

MediaMTX JWT 지원 여부
----------------------
공식 지원 확인됨 (mediamtx.org/docs/usage/authentication).

방식 A — JWT 직접 검증 (authMethod: jwt)
  - MediaMTX가 API Server의 JWKS 엔드포인트에서 공개키를 가져와 자체 검증
  - mediamtx.yml 설정 예시:
      authMethod: jwt
      authJWTJWKS: https://your-api-server/api/auth/jwks.json
  - 클라이언트 전달 방법:
      Authorization: Bearer <JWT>  (권장)
      또는 ?jwt=<JWT> 쿼리 파라미터 (비권장 — 로그 노출 위험)

방식 B — HTTP 콜백 (authMethod: http)
  - MediaMTX가 클라이언트 요청마다 API Server에 인증 위임
  - 더 단순하지만 API Server에 요청 부하 발생

주의사항:
  - JWKS는 시작 시 한 번만 가져오므로 키 교체 시 MediaMTX 재시작 필요
  - v1.8.0 이전에는 Authorization 헤더 JWT 무시 버그 있음 (v1.8.0에서 수정)

업계 사례
---------
동일 패턴을 사용하는 서비스:
  - AWS IVS: "Playback Token" (ES384 JWT)
    https://docs.aws.amazon.com/ivs/latest/LowLatencyUserGuide/private-channels-workflow.html
  - Cloudflare Stream: "Signed URL" (RS256 JWT)
    https://developers.cloudflare.com/stream/viewing-videos/securing-your-stream/
  - Mux: "Signed URL"
    https://www.mux.com/articles/securing-video-playback-with-signed-urls

패턴 명칭: "Signed URL", "Playback Token", "Token Vending Architecture" 등
           단일 공식 명칭은 없으나 업계 표준 패턴으로 확인됨.

다이어그램 반영 방법
--------------------
2-2_overall_system_conf.jpg에서 MediaMTX → Client App 화살표는
데이터 경로(스트리밍)를 표현한다.
JWT 토큰 발급(제어 경로)은 Client App ↔ API Server 화살표에 포함된다.
세부 흐름은 2.3 시나리오(라이브 스트리밍 항목)에 기술되어 있다.
