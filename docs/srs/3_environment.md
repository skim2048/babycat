# 3. 환경 (Environment)

## 3.1 운영 환경 (Operating Environment)

## 3.2 제품 설치 및 설정 (Product Installation and Configuration)

- 시스템은 컨테이너 이미지의 빌드와 기동으로 설치되며, 설치 전에 운영자가 다음을 설정해야 한다.
  - 외부 도달 가능한 IP 주소. WebRTC가 이 주소를 접속 후보로 광고하므로, 설정되지 않으면 외부 노드의 라이브 스트리밍 접속이 실패한다.
  - JWT 서명 비밀키. 운영 환경에서 기본값을 그대로 사용해서는 안 된다.
  - 분석에 사용할 후보 VLM 모델의 목록.
- 최초 기동 시 VLM 모델의 사전 컴파일이 수행되며 모델당 수 분에서 수십 분이 소요된다. 컴파일 결과는 캐시되어 이후 기동에서 재사용된다.
- 최초 기동 시 초기 관리자 계정이 1회 생성되며, 최초 로그인 시 비밀번호 변경이 요구된다.

구체적인 설치 명령과 설정 항목의 목록은 설치 가이드에서 다룬다.

## 3.3 배포 환경 (Distribution Environment)

- **마스터 구성**: 소스 저장소를 마스터로 삼으며, 사전 빌드된 이미지 레지스트리는 두지 않는다. 마스터에는 소스코드, 컨테이너 빌드 파일, 설정 템플릿, 설계 문서가 포함된다.
- **배포 방법**: 대상 장비에서 저장소를 받아 현장에서 이미지를 빌드한다.
- **설치 방법**: 위 제품 설치 및 설정을 따른다.
- **패치 및 업데이트**: 저장소를 갱신한 뒤 재빌드 및 재기동하며, 자동 업데이트는 제공하지 않는다.

## 3.4 개발 환경 (Development Environment)

- ***Engine***은 하드웨어 가속기와 Jetson 플랫폼 라이브러리에 의존하므로 Jetson 실기에서만 개발할 수 있다.
- ***Gateway***는 일반 PC에서 개발할 수 있다.
- 개발에는 Git과 Docker가 필요하다.

## 3.5 테스트 환경 (Test Environment)

- 단위 테스트를 수행하며, GStreamer 및 하드웨어에 의존하는 테스트는 Jetson 실기에서만 가능하다.
- 통합 테스트는 Jetson 실기에서 수행하며, ***Video Source***는 실제 카메라 또는 사전 녹화 비디오를 RTSP로 송출하는 대체 소스로 구성한다. 대체 소스는 동일한 입력을 반복 재현할 수 있어 프롬프트 및 이벤트 키워드 튜닝에 사용한다.
- 권장사양 환경의 테스트 확보 여부는 작성을 보류한다.

## 3.6 형상 관리 (Configuration Management)

- **시스템**: Git과 GitHub을 사용한다.
- **베이스라인 포함 산출물**: 소스코드, 설계 문서, 컨테이너 빌드 파일, 설정 템플릿.
- **형상 관리 제외 대상**: 비밀키와 런타임 상태는 저장소에 포함하지 않는다.
- **브랜치 정책**: 안정 브랜치를 유지하고, 작업은 별도 브랜치에서 수행한 뒤 병합한다.
- **태깅 정책**: 버전 릴리스 시 태그를 부여한다.
