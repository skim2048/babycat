# 6. Non-functional Requirements; 비기능 요구사항

식별자 번호는 5장의 `NFR`과 이어진다.

## 6.1 Safety; 안전성 요구사항

- `NFR-009` Babycat은 모니터링 보조 도구이며, VLM의 한계로 이벤트 미탐지·오탐지가 발생할 수 있다. 시스템은 안전사고 예방을 보장하지 않으며, 이 한계는 사용자에게 고지되어야 한다. 고지 방식(문서, Client App 표시 등)은 작성 보류.
- `NFR-010` 저장 공간 확보를 위한 클립 자동 삭제(`FR-024`)로 오래된 이벤트 클립이 소실될 수 있다. 이 동작은 문서화되어야 하며, 삭제 이력은 로그로 남겨야 한다.

## 6.2 Security; 보안 요구사항

- `NFR-011` 시스템은 동일 계정에 대해 10분 내 10회 로그인 실패 시 해당 계정의 로그인을 30분간 차단해야 한다.
- `NFR-012` 비밀번호는 솔트를 적용한 PBKDF2-HMAC-SHA256(반복 100,000회 이상)으로 해시하여 저장해야 한다. 원문은 저장하지 않는다.
- `NFR-013` JWT는 HMAC-SHA256으로 서명하며, 서명 비밀키(`JWT_SECRET`)는 환경 변수로 주입한다. 기본값을 운영 환경에서 사용해서는 안 된다. 기본값 사용 시의 시스템 동작(기동 거부 또는 경고)은 작성 보류.
- `NFR-014` 리프레시 토큰은 해시로만 저장하고, 사용 시 회전하며, 비밀번호 변경 시 전부 폐기해야 한다(7.1 참조).
- `NFR-015` 카메라 프로필의 비밀번호는 API 응답에 원문이 포함되지 않아야 한다.
- `NFR-016` 전송 계층 암호화(HTTPS/TLS)의 적용 여부와 적용 범위는 작성 보류. 미적용 시 신뢰된 네트워크(LAN/VPN) 내 운영을 전제 조건으로 명시해야 한다.

## 6.3 System Attributes; 소프트웨어 시스템 특성

### 6.3.1 Availability; 가용성

- `NFR-017` 시스템은 24시간 무인 동작을 전제로 한다.
- `NFR-018` 컨테이너가 비정상 종료된 경우 자동으로 재시작되어야 한다(컨테이너 런타임의 재시작 정책 이용).
- `NFR-019` App Server는 기동 시 미디어 서버가 준비되지 않았으면 지수 백오프로 재시도해야 한다(`FR-012`).

### 6.3.2 Maintainability; 유지보수성

- `NFR-020` 시스템은 역할별 컨테이너(2.2의 컴포넌트 단위)로 분리하여 독립적으로 빌드·교체 가능해야 한다.
- 로그 수집·진단 기능에 대한 요구사항은 작성 보류.

### 6.3.3 Portability; 이식성

- Babycat은 Jetson Platform 전용이다. 타 플랫폼 이식은 범위 밖이며(2.7 참조), 이식성 요구사항을 두지 않는다.

### 6.3.4 Reliability; 신뢰성

- `NFR-021` 장기 무인 구동 보장 기간(예: 30일 무중단) 및 측정 기준은 작성 보류.

## 6.4 Logical Database Requirements; 데이터베이스 요구사항

- 데이터베이스는 SQLite 단일 파일(`/data/db/babycat.db`)을 사용하며 WAL 모드로 동작한다.
- 저장 엔티티: 사용자 계정, 리프레시 토큰, 이벤트 이력. (디바이스 토큰은 차기 버전 — 2.7 참조.)
- 클립 파일과 메타데이터는 데이터베이스가 아닌 파일시스템(`/data/{YYYY}/{MM}/`)에 저장한다(2.2 Storage 참조).
- 이벤트 이력의 보존 기간·상한은 작성 보류.

## 6.5 Business Rules; 비즈니스 규칙

영상 정보가 포함되므로 개인영상정보 관련 법규(개인정보보호법 등)의 적용 여부 검토가 필요하다. 작성 보류.

## 6.6 Design and Implementation Constraints; 설계와 구현 제한사항

2.6의 종속관계를 제약사항 식별자로 확정한다.

- `CON-001` 시스템은 Jetson Platform(Orin NX 이상 급) 및 JetPack 6.2를 대상으로 한다.
- `CON-002` VLM 추론은 NanoLLM 스택을 사용한다.
- `CON-003` 입력 비디오 코덱은 H.264로 한정한다.
- `CON-004` 시스템은 Docker Compose 기반 컨테이너로 배포한다.
- 코딩 컨벤션 등 개발 규칙은 작성 보류.

## 6.7 Memory Constraints; 메모리 제한사항

- `NFR-022` 시스템 전체(VLM 포함)는 Jetson Board의 16GB 통합 메모리 내에서 동작해야 한다.
- `NFR-023` VLM은 16GB 메모리에 적재 가능한 3B급 모델로 한정하며, 복수 모델의 동시 적재는 금지한다. 모델 전환은 캐시 기반 재적재로만 수행한다.

## 6.8 Operations; 운영 요구사항

- 시스템은 상시 무인 운영을 기본으로 하며, 스토리지 용량 관리는 자동으로 수행된다(`FR-024`).
- 운영자의 정기 작업(데이터베이스 백업, 로그 점검 등) 정의는 작성 보류.

## 6.9 Site Adaptation Requirements; 사이트 적용 요구사항

- 설치 사이트에서 Jetson Board가 RTSP Source와 동일 네트워크에 도달 가능해야 한다.
- 4.5의 포트가 방화벽에서 개방되어야 한다.
- WebRTC 사용 시 `HOST_IP`(외부 도달 가능 IP)가 사이트 네트워크 구성에 맞게 설정되어야 한다(3.2).

## 6.10 Internationalization Requirements; 다국어 지원 요구사항

이 시스템에는 다국어 지원 요구사항이 없다. Babycat은 사용자 인터페이스를 제공하지 않으며(4.2), 다국어화는 Client App의 책임이다.

## 6.11 Unicode Support; 유니코드 지원

- `NFR-024` API 요청·응답, 클립 메타데이터(json), 데이터베이스의 모든 텍스트는 UTF-8로 인코딩해야 한다.

## 6.12 64bit Support; 64비트 지원

시스템은 aarch64(64비트) 전용이다. JetPack/L4T가 64비트 운영체제이므로 별도 요구사항을 두지 않는다.

## 6.13 Certification; 제품 인증

이 시스템에는 제품 인증 요구사항이 없다.
