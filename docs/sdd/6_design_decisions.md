| ID | 결정 | 대안 | 근거 | 상태 |
|---|---|---|---|---|
| D-A | ***Gateway***를 제어 평면의 단일 외부 진입점으로 둔다. ***Engine***·***Media***·***Storage***는 내부 전용이며 클라이언트가 직접 도달하지 못한다. 라이브 미디어만 데이터 평면 예외로 ***Client App*** ↔ ***Media*** 직결을 허용하되 토큰으로 게이트한다. | 내부 컴포넌트를 부분 노출(현행 8080 호스트 게시 유지) | 인증·인가를 ***Gateway*** 한 곳에 모아 공격면을 단일화하고, 클라이언트가 내부 토폴로지를 알 필요가 없게 한다. 미디어 직결은 대용량 스트림 중계 시의 부하·지연을 피하기 위한 예외다. | 결정 |

## 6.1 결정 보충; Notes

결정 D-A·D-B에 딸린 세부와 합의된 트레이드오프를 기록한다.

- **두 가지 집행 메커니즘(D-B).** 라이브 스트리밍은 프로토콜에 따라 무인가 차단 방식이 다르다. WebRTC는 "***Gateway*** 경유 시그널링 + ***Media*** 시그널링 포트 망 격리"로, HLS는 "***Media*** → ***Gateway*** HTTP 콜백 토큰 검증"으로 집행한다. HLS는 요청과 미디어가 분리되지 않아(세그먼트 GET이 곧 미디어) 시그널링만 경유시킬 수 없기 때문이다.
- **합의된 트레이드오프(D-B).** 세션 토큰을 재사용하므로 스트리밍 전 별도 토큰 발급 왕복이 없다. 대신 HLS는 토큰이 URL 쿼리스트링에 실리기 쉬워 세션 토큰이 로그·히스토리에 노출될 위험이 있다. 노출 차단이 필요해지면 단기·경로한정 재생 토큰을 별도 발급하는 방식(왕복 1회 추가)으로 전환한다.
- **구현 확인사항(D-B).** MediaMTX 콜백이 토큰을 어느 필드로 전달하는지(Authorization 헤더 vs `?token=` 쿼리)는 구현 시 MediaMTX 문서로 확인한다. 설계 결정 자체에는 영향이 없다.
- **모니터링 피드(D-A→D-D).** 단일 진입점을 완성하려면 모니터링 피드(SRS 기능 7, SSE/MJPEG)도 ***Gateway***를 경유해야 한다. 이 피드가 프록시 표면에 포함되는지는 D-D에서 확정한다.

## 6.2 조사 기록; Findings

결정에 앞서 확인된 사실을 기록한다.

### F-1. Engine 8080 호스트 게시 (D-A, 해소됨)

***Engine***의 8080 포트가 `docker-compose.yml`에서 호스트에 게시되어 있어, 클라이언트가 ***Gateway***를 우회하여 ***Engine***에 직접 접근하는 경로가 외부에 열려 있다. 이는 의도된 운영 설계가 아니라 **디버그 잔재**이다.

- **근거(이력).** 8080 게시는 커밋 `be96082`("디버그 대시보드…")에서 주석 `# Debug web UI`와 함께 도입되었다. ***Engine***이 직접 서빙하는 디버그 대시보드(`/stream` MJPEG, `/events` SSE 등)에 ***Gateway***를 거치지 않고 붙기 위한 것이었다. 이후 커밋 `b11177a`에서 주석만 `# App HTTP API`로 변경되었고, **호스트 게시는 회수되지 않은 채 상존한다.**
- **정상 대비.** 같은 파일의 MediaMTX RTSP 8554는 의도적으로 `ports`에 넣지 않아 내부망(`mtx:8554`) 전용이다. 8080의 정당한 소비자도 ***Gateway*** 하나뿐이며 내부 주소(`app:8080`)로 충분하다.
- **해소.** D-A가 단일 진입점(SRS 2.6)으로 결정되고 D-B의 스트리밍 라우팅이 확정되어 제거 순서 의존성이 풀렸다. 따라서 **8080 호스트 게시는 제거 대상으로 확정**한다. `docker-compose.yml`에서 8080을 `ports`에서 내려 내부 전용(`app:8080`)으로 둔다.
