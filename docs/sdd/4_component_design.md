# 4. 컴포넌트 설계 (Component Design)

각 절은 책임, 내부 구조, 외부에 노출하는 인터페이스, 내부 상태의 순서로 서술한다.

컴포넌트의 이름은 SRS를 따르며, 서비스와 디렉터리는 같은 이름을 소문자로 쓴다.

|컴포넌트|서비스|이미지|기반|
|---|---|---|---|
|***Gateway***|`gateway`|`babycat-gateway`|FastAPI|
|***Engine***|`engine`|`babycat-engine`|표준 라이브러리 HTTP 서버, GStreamer, NanoLLM|
|***Media***|`media`|`bluenviron/mediamtx`|MediaMTX|
|***Storage***|-|-|호스트 디렉터리 마운트|

## 4.1 Gateway

### 책임

외부 요청의 인증과, 계정·이벤트 발생 이력·비디오 클립에 대한 처리를 맡는다. 장면 분석과 ***Video Source***에 관한 요청은 스스로 처리하지 않고 ***Engine***에게 전달한다.

### 내부 구조

|모듈|책임|
|---|---|
|`main`|요청 경로의 정의와 처리|
|`auth`|자격증명 검증, 토큰의 발급과 회전과 폐기, 로그인 시도 제한|
|`database`|데이터베이스 연결과 스키마 초기화|
|`schemas`|요청과 응답의 형식 정의|
|`engine_proxy`|***Engine***으로의 요청 전달과 응답 정규화|
|`clip_support`|클립 파일 경로의 해석과 부분 전송 처리|

비밀번호는 솔트와 함께 PBKDF2-HMAC-SHA256을 100,000회 적용하여 저장한다(SRS `NFR-012`). 토큰은 HMAC-SHA256으로 서명하며 서명 비밀키는 외부에서 주입받는다(SRS `NFR-013`).

로그인 시도 제한은 계정별 실패 횟수를 세어 임계에 이르면 일정 시간 로그인을 막는다(SRS `FR-007`). 다만 이 계수를 어디에 둘지는 결정을 요한다. 프로세스 메모리에 두면 컨테이너가 재시작될 때마다 초기화되므로, 재시작을 반복하여 제한을 회피할 여지가 생긴다. (로그인 시도 계수의 보존 방식 명세는 작성을 보류한다.)

### 노출하는 인터페이스

***Client App***에게 HTTP를 제공한다. 경로별 상세는 §6.1에서 정한다.

***Engine***에게 요청을 전달할 때는 들어온 인증 헤더를 그대로 실어 보낸다. 따라서 ***Engine***도 토큰을 검증할 수 있어야 하며, 두 컴포넌트가 같은 서명 비밀키를 공유한다.

***Engine***의 응답을 그대로 돌려주지 않고 한 번 정규화한다. ***Video Source*** 프로필의 비밀번호는 값을 지우고 설정 여부만 남겨 반환한다(SRS `FR-013`). ***Engine***이 응답하지 않거나 서버 오류를 돌려주면 502로 바꾸어 응답한다.

### 내부 상태

데이터베이스 외에 유지하는 상태는 로그인 시도 계수뿐이다. 그러므로 ***Gateway***는 데이터베이스를 제외하면 상태를 갖지 않으며, 재시작으로 잃는 것이 없다.

## 4.2 Engine

### 책임

장면 분석과 이벤트 감지, ***Video Source*** 프로필의 관리와 PTZ 제어, 그리고 이벤트 구간의 클립 기록을 맡는다. 하드웨어 가속기와 Jetson 플랫폼 라이브러리에 의존하는 유일한 컴포넌트다.

### 내부 구조

|모듈|책임|
|---|---|
|`main`|기동 절차, 파이프라인 구성, 추론 루프, 클립 기록|
|`server`|HTTP 요청 처리|
|`camera`|프로필의 저장과 적용, ***Media***의 스트림 소스 갱신|
|`ptz`|ONVIF를 이용한 PTZ 제어와 홈 위치 관리|
|`state`|공유 실행 상태와 상태 전달|
|`vlm_worker`|VLM 자식 프로세스의 생성, 추론 요청, 종료|
|`clip_storage`|저장 공간 확인과 오래된 클립의 삭제|
|`hardware`|하드웨어 상태 측정|

### 실행 단위

***Engine***은 하나의 주 프로세스와 하나의 자식 프로세스로 이루어진다.

주 프로세스는 다음 실행 단위를 가진다.

|실행 단위|역할|수명|
|---|---|---|
|주 스레드|GStreamer 이벤트 루프|상시|
|HTTP 서버 스레드|요청 수신|상시|
|요청 처리 스레드|요청 하나의 처리|요청마다 생성|
|추론 스레드|프레임 취득, 추론 요청, 키워드 대조|상시, 하나|
|워치독 스레드|파이프라인 정지의 감지와 재기동|상시|
|세그먼트 기록 스레드|클립 재구성을 위한 비디오 조각의 기록|상시|
|PTZ 상태 폴링 스레드|이동 여부의 갱신|상시|
|클립 저장 스레드|이벤트 구간 클립의 생성|이벤트마다 생성|

VLM은 주 프로세스가 아니라 자식 프로세스에서 동작한다. 모델 실행 엔진이 모델을 놓을 때 CUDA와 TensorRT가 점유한 메모리를 프로세스 안에서 온전히 반환하지 못하기 때문이다. Jetson은 GPU와 CPU가 하나의 메모리를 나누어 쓰므로, 반환되지 않은 메모리는 전환할 때마다 누적되어 16GB 한계를 잠식한다. 자식 프로세스를 종료하면 운영체제가 이를 모두 회수하므로, 모델 전환을 프로세스의 교체로 구현한다. 이것이 SRS `NFR-023`이 요구한 "메모리 반환 후 재적재"의 실현 수단이다.

주 프로세스는 모델 실행 엔진을 적재하지 않으며 CUDA를 직접 다루지 않는다. 두 프로세스는 유닉스 도메인 소켓으로 통신한다.

이 구조로 §3.4가 요구한 세 가지 분리가 성립한다. 추론은 자식 프로세스에서 일어나므로 주 프로세스를 막지 않고, 제어 요청은 요청마다 생성되는 스레드가 처리하므로 추론 스레드를 기다리지 않으며, 클립 저장은 이벤트마다 별도 스레드로 떼어내므로 다음 추론을 늦추지 않는다.

### 장면 분석 파이프라인

비디오는 다음 순서로 처리된다.

```
rtspsrc → rtph264depay → h264parse → nvv4l2decoder → nvvidconv → videorate → appsink
```

디코딩은 하드웨어 디코더가 맡는다. `videorate`가 프레임 비율을 추론에 필요한 수준으로 낮추므로, 스트림의 원래 프레임 비율과 무관하게 추론 부하가 일정하게 유지된다.

`appsink`의 콜백 스레드는 받은 프레임을 고정 크기 링 버퍼에 넣는다. 추론 스레드는 링 버퍼에서 최근 프레임 여러 장을 한 번에 꺼내 VLM에 입력한다. 링 버퍼를 두는 이유는 두 가지다. 프레임 취득과 추론의 속도가 다르므로 둘을 분리해야 하고, VLM이 한 장이 아니라 여러 장을 함께 보아야 움직임을 판단할 수 있기 때문이다.

추론 루프는 매 반복의 시작에서 모델 전환 요청을 확인한다. 전환은 추론과 추론 사이에서만 일어나며 생성 도중에는 일어나지 않는다.

PTZ가 이동하는 동안에는 추론을 건너뛴다. 이동 중의 화면은 흐려지고 장면의 의미도 바뀌는 중이므로, 판정의 근거로 삼기에 적절하지 않기 때문이다.

파이프라인의 상세한 흐름은 §7.2에서 다룬다.

### 이벤트 판정과 기록

추론이 만든 텍스트를 소문자로 바꾸고, 설정된 이벤트 키워드가 그 안에 들어 있는지 확인한다. 하나라도 들어 있으면 이벤트로 판정한다.

이벤트로 판정하면 해당 구간의 클립을 만든다. 클립은 이벤트가 판정된 시점 이후만이 아니라 그 앞의 구간까지 담아야 한다. 이벤트를 일으킨 장면은 판정보다 먼저 지나갔기 때문이다. 따라서 스트림을 짧은 조각으로 미리 기록해 두었다가, 이벤트가 나면 앞뒤 조각을 이어 붙여 하나의 클립을 만든다. 이 조각은 휘발성 저장 공간에 두고 일정 시간이 지나면 지운다.

클립을 쓰기 전에 남은 저장 공간을 확인한다. 임계에 못 미치면 오래된 클립부터 지워 공간을 확보한다(SRS `FR-033`). 한 번에 지우는 개수에는 상한을 둔다.

이벤트 발생 이력을 데이터베이스에 남기는 일은 SRS §2.3 (4)가 ***Engine***의 책임으로 정하고 있으나, 데이터베이스는 ***Gateway***가 소유한다. ***Engine***이 데이터베이스를 직접 열면 두 컴포넌트가 같은 파일에 쓰게 되고, ***Gateway***에게 요청하면 §3.3이 정한 의존의 방향이 뒤집힌다. 어느 쪽도 그대로 받아들일 수 없으므로 별도의 판단이 필요하다. (이벤트 발생 이력의 기록 경로 명세는 작성을 보류한다.)

### 노출하는 인터페이스

***Gateway***에게 HTTP를 제공한다. 프로필 조회와 적용, 프롬프트와 이벤트 키워드의 설정, PTZ 제어, VLM 전환, 클립 목록과 삭제, 그리고 분석 상태의 전달이 여기에 속한다.

분석 상태는 요청에 대한 응답으로 돌려줄 수 없다. 상태가 바뀌는 시점을 ***Client App***이 알 수 없기 때문이다. 따라서 연결을 유지한 채 서버가 갱신을 밀어 보내는 방식을 사용한다. 이것이 §3.3에서 남긴 "상태를 확인할 별도의 경로"에 해당한다.

### 내부 상태

***Engine***은 다른 컴포넌트와 달리 상당한 실행 상태를 메모리에 지닌다. 현재 프롬프트와 이벤트 키워드, 적재된 VLM과 그 상태, 파이프라인의 구동 여부, 최근 추론 결과, 링 버퍼의 내용이 그것이다.

이 가운데 프롬프트와 이벤트 키워드와 프로필은 재시작 후에도 남아야 하므로 ***Storage***에 파일로 보존한다. 나머지는 재시작으로 잃어도 무방하다.

## 4.3 Media

### 책임

***Video Source***의 스트림을 받아 내부에 재배포하고, 소비자가 요구하는 프로토콜로 변환하여 전달한다.

### 내부 구조

MediaMTX를 그대로 사용하므로 내부 구조를 `Babycat`이 설계하지 않는다. `Babycat`이 관여하는 것은 설정 파일과, 구동 중에 보내는 제어 요청뿐이다.

기성 소프트웨어를 쓰는 대신 얻는 것과 잃는 것이 분명하다. RTSP 수신과 HLS·WebRTC 변환을 직접 구현하지 않아도 되나, 그 동작을 세밀하게 바꿀 수 없고 문제가 생겼을 때 안을 들여다보기 어렵다. 프로토콜 변환은 `Babycat`이 풀려는 문제가 아니므로 이 교환은 받아들일 만하다.

### 스트림 소스의 지정

***Media***는 어느 ***Video Source***로부터 스트림을 받을지 설정으로 알며, 이 설정은 구동 중에 바뀔 수 있어야 한다. ***User***가 프로필을 바꾸면 ***Engine***이 ***Media***의 제어 API에 요청하여 스트림 소스를 갱신한다.

전송은 TCP로 지정한다. UDP는 손실을 허용하므로 화면이 깨질 수 있고, `Babycat`은 실시간성보다 프레임의 온전함을 중시하기 때문이다.

### 노출하는 인터페이스

|대상|프로토콜|용도|
|---|---|---|
|***Engine***|HTTP|스트림 소스의 조회와 갱신|
|***Engine***|RTSP|재배포된 스트림의 소비|
|***Client App***|HLS|라이브 비디오 재생|
|***Client App***|WebRTC|저지연 라이브 비디오 재생|

WebRTC는 연결 후보 주소를 상대에게 알려야 연결이 성립한다. `Babycat`이 구동되는 장비의 주소는 설치 대상마다 다르므로, 이 값은 설치 시점에 외부에서 주입받는다(SRS §6.9).

### 내부 상태

스트림 소스의 설정과 진행 중인 세션을 지닌다. 재시작하면 세션이 끊기고 ***Client App***은 다시 접속해야 한다. 스트림 소스 설정은 ***Engine***이 다시 보내야 하므로, ***Media***가 재시작한 뒤 ***Engine***이 이를 감지하여 설정을 복구할 수 있어야 한다. (***Media*** 재시작 시의 설정 복구 명세는 작성을 보류한다.)

## 4.4 Storage

### 책임

데이터베이스, 비디오 클립, 사이드카 메타데이터, 설정, VLM 컴파일 캐시를 담는다. 스스로 동작하지 않는다.

### 구성

|영역|내용|접근하는 컴포넌트|
|---|---|---|
|데이터베이스|계정, 리프레시 토큰, 이벤트 발생 이력|***Gateway***|
|클립|비디오 클립과 사이드카 메타데이터|***Engine***이 쓰고 ***Gateway***가 읽고 지움|
|설정|***Video Source*** 프로필, PTZ 홈 위치|***Engine***|
|모델 캐시|VLM의 사전 컴파일 결과|***Engine***|

클립에는 데이터베이스를 거치지 않는 접근 경로가 있다. ***Engine***이 파일을 쓰고 ***Gateway***가 같은 영역을 마운트하여 목록을 만들고 내용을 읽는다. 두 컴포넌트를 잇는 것은 호출이 아니라 파일 이름의 규약이다. 이 규약이 깨지면 ***Gateway***가 ***Engine***이 쓴 파일을 찾지 못하며, 그 실패는 어느 쪽의 오류로도 드러나지 않는다. 규약의 상세는 §5.3에서 정한다.

모델 캐시는 재생성이 가능하나 그 비용이 크다. 사전 컴파일에 모델당 수 분에서 수십 분이 걸리므로(SRS §3.2), 컨테이너를 다시 빌드하더라도 이 영역은 보존되어야 한다.

이벤트 구간을 재구성하기 위한 비디오 조각은 ***Storage***에 두지 않는다. 짧은 주기로 쓰고 지우기를 반복하므로 보조기억장치의 수명을 깎으며, 재시작 후에 남아 있을 이유도 없기 때문이다. 이 조각은 휘발성 영역에 둔다.

### 동시 접근

§3.3이 지적한 ***Gateway***와 ***Engine***의 동시 쓰기는, 위 구성에서는 발생하지 않는다. 데이터베이스는 ***Gateway***만, 설정은 ***Engine***만 쓰기 때문이다. 클립 영역은 양쪽이 쓰나 ***Engine***은 새 파일을 만들 뿐이고 ***Gateway***는 기존 파일을 지울 뿐이므로, 같은 파일을 두고 다투지 않는다.

다만 ***Gateway***가 지우려는 순간 ***Engine***이 아직 쓰고 있는 파일이라면 충돌한다. 기록이 끝나지 않은 파일을 목록에서 제외할 기준이 필요하다. (기록 중인 클립의 식별 방식 명세는 작성을 보류한다.)
