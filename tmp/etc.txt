1.
Babycat 내부의 세 컴포넌트의 이름을 아래와 같이 변경할 것이다.
  API Server      -> Gateway
  App Server      -> Engine
  MediaMTX Server -> Media
(Storage는 자원이므로 이름을 유지한다.)

2.
Client App의 WebRTC 요청과 서비스는 아래 경로를 거치게 한다.
  요청(시그널링-제어평면): Client App -> Gateway -> Media
  수신(스트리밍-데이터평면): Media -> Client App

  p.s.
  - WebRTC는 시그널링(요청·협상)과 미디어(실제 스트림)가 분리되므로,
    시그널링만 Gateway가 경유하고 미디어는 Media가 직접 송출하는 위상이 성립한다.
  - 무인가 차단은 토큰 검증이 아니라 "Gateway 경유 + 망 격리"로 이룬다.
    즉 Media의 시그널링 창구는 Gateway에서만 도달 가능하도록 격리하며,
    미디어 경로는 시그널링에서 협상된 ICE 세션 자격으로 보호된다.

3.
Client App의 HLS 요청과 서비스는 아래 경로를 거치게 한다.
  요청·수신(데이터평면): Client App <-> Media

  p.s.
  - HLS는 요청과 스트림이 분리되지 않는다. 재생목록(.m3u8)·세그먼트 GET이 곧 미디어 전송이므로,
    HLS 트래픽 전체가 데이터평면이며 Media에 직결한다. (WebRTC와 달리 별도 제어평면이 없다.)
  - 따라서 매 요청을 Gateway로 경유시키지 않는다. 그렇게 하면 미디어를 Gateway가 중계하게 되어
    저지연·무중계 목표가 깨지기 때문이다.
  - Media는 Gateway가 발급한 토큰이 없는 요청을 무시한다.
    클라이언트의 Media 직결은 "허가된 직결"이지 Gateway 우회가 아니다.
  - 이때 사용하는 토큰은 스트리밍 시점에 따로 발급받지 않고,
    로그인 시점에 Gateway가 발급한 세션 토큰을 재사용한다.
    (스트리밍 전 별도의 토큰 발급 왕복을 두지 않기 위한 선택이다.)
  - 트레이드오프: 세션 토큰 재사용은 Gateway 왕복이 없어 단순하나,
    HLS는 토큰이 URL 쿼리스트링에 실리기 쉬워 세션 전체 권한 토큰이 로그·히스토리에 노출될 수 있다.
    노출을 차단하려면 단기·경로한정 재생 토큰을 별도 발급해야 하며,
    그 경우 스트리밍 전 Gateway 왕복 1회가 생겨 "요청은 Gateway를 거치지 않는다"가 깨진다.

4.
위 1~3의 결정에 따라 미결 항목 D-A와 D-B를 해결한다. (SDD 6장으로 이관 완료)

D-A (단일 진입점):
  - Gateway를 제어 평면의 단일 외부 진입점으로 둔다.
  - Engine·Media·Storage는 내부 전용이며 클라이언트가 직접 도달하지 못한다.
  - 라이브 미디어만 데이터평면 예외로 Client App <-> Media 직결을 허용하되 토큰으로 게이트한다.
  - F-1(Engine 8080 호스트 게시)은 제거 대상으로 확정. docker-compose.yml에서 8080을 내부 전용으로 둔다.

D-B (스트리밍 인증):
  - 미디어 접근은 Gateway가 발급한 토큰으로 게이트한다.
  - WebRTC: 시그널링 Gateway 경유 + Media 시그널링 포트 망 격리 (Media 자체 검증 없음).
  - HLS: Media가 매 요청을 Gateway로 HTTP 콜백하여 토큰 검증 (방식 B). 즉시 철회 가능.
  - 토큰: 별도 발급 없이 로그인 세션 토큰 재사용.
  - 대안(방식 A: Media 자체 검증 RS256+JWKS)은 현재 HS256 토큰과 맞지 않아 보류.
  - 트레이드오프: HLS 토큰의 쿼리스트링 노출. 차단 필요 시 단기·경로한정 재생 토큰으로 전환(왕복 1회 추가).
  - 구현 확인: MediaMTX 콜백의 토큰 전달 필드(Authorization 헤더 vs ?token= 쿼리)는 문서로 확인.

5.
남은 미결: 모니터링 피드(기능 7, SSE/MJPEG)도 Gateway를 경유해야 단일 진입점이 완성된다.
이 피드가 프록시 표면에 포함되는지는 D-D(Engine-Gateway 내부 계약)에서 확정한다.
